GIN Authentication Microservice

Enterprise-grade Authentication API with 2FA, JWT, and Device Tracking

API Documentation

Interactive API explorer for the GIN Authentication Microservice. Use this interface to explore, test, and understand all available endpoints.

Key Features

  • JWT Authentication: Secure token-based authentication with 15-minute access tokens and 7-day refresh tokens
  • 2FA Support: Two-Factor Authentication using TOTP (Google Authenticator, Authy) and backup login codes
  • Password Security: Bcrypt+pepper hashing, complexity validation, breach checking, and password history
  • Device Tracking: Geolocation-based security, device identification, and session management
  • Rate Limiting: Brute force protection with CAPTCHA and 15-minute blocks
  • Role-Based Access: Admin, Moderator, User, and Staff roles with different permissions

Getting Started

  • Base URL: http://localhost:8080
  • Authentication: Use Bearer token in Authorization header
  • Content-Type: All requests use application/json
  • Response Format: Success returns data object, errors return {"error": "message"}

Security Headers

  • Authorization: Authorization: Bearer YOUR_ACCESS_TOKEN
  • Cookies: Refresh token automatically stored in HTTP-only cookie
  • CSRF: Protected via refresh token pattern (not needed)

Common HTTP Status Codes

  • 200: Success
  • 202: Accepted (verification needed)
  • 400: Bad Request (invalid input)
  • 401: Unauthorized (invalid token/credentials)
  • 403: Forbidden (insufficient permissions)
  • 429: Too Many Requests (rate limited)
  • 500: Server Error

Authentication Flow Example

  1. POST /user/signup with name, email, password → Get verification email
  2. POST /user/verify_signup_email with code → Account created
  3. POST /user/login with credentials → Get verification code
  4. POST /user/verify-email with code → Get access & refresh tokens
  5. Use access token in all authenticated requests
  6. When access token expires, POST /user/refresh to get new token

User Roles

  • user: Regular user with basic permissions
  • moderator: Can perform moderation actions
  • admin: Full system access
  • staff: Staff-level permissions
